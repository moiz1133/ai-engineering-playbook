# Semantic and Structure-Aware Chunking

Rather than slicing text at arbitrary fixed-length boundaries, structure-aware chunking splits documents at points that already carry meaning: paragraph breaks, Markdown headings, HTML tags, function or class boundaries in code, or sentence boundaries via a sentence tokenizer. The goal is for each resulting chunk to correspond to one coherent unit of thought, rather than an arbitrary token-count window that might bisect a sentence or a logical block.

A common approach for Markdown or HTML documents is header-aware splitting: walk the document's heading hierarchy (H1, H2, H3, ...) and treat each section as a candidate chunk, optionally prepending the heading path (e.g., "Chapter 3 > Installation > Prerequisites") to the chunk text so the embedding captures the section's place in the document's structure, not just its raw content. This tends to produce chunks whose embeddings are noticeably more precise than fixed-size windows of the same document, since each chunk maps to something a human would also call "one topic."

A more computationally expensive variant is true semantic chunking: embed individual sentences, then walk through the document measuring the embedding similarity between consecutive sentences, and insert a chunk boundary wherever similarity drops sharply (indicating a topic shift). This adapts chunk boundaries to the actual content rather than the document's markup, which helps on plain-text sources with no reliable heading structure, at the cost of requiring an embedding call per sentence during ingestion.

Structure-aware chunking's main limitation is that it depends on the source having exploitable structure in the first place — a wall of unstructured prose with no headings and no clear topic-sentence pattern gets little benefit over fixed-size chunking, and the resulting chunk sizes become unpredictable, which can cause chunks that are individually too large or too small for the embedding model to represent well. Many production pipelines combine both: split by structure first, then apply a fixed-size fallback split to any resulting section that's still too long.
